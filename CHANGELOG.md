# Changelog

## 1.18.0
- Добавлен модуль `storage` с минимальным ATA PIO transport-layer API (`storage_init`, `storage_read_sectors`) для чтения LBA-секторов через единый блочный контракт внутри ядра.
- В driver-stage ядро теперь выполняет тестовое чтение фиксированного сектора `LBA 0`, проверяет сигнатуру загрузочного сектора `0x55AA` и публикует результат как runtime-диагностику.
- Footer UI расширен диагностикой storage-path (`DISK READY/SIG`, `DISK LBA`), а `README.md`, `DEVELOPMENT_PLAN.md` и `docs/ARCHITECTURE.md` обновлены под начало этапа D.1 roadmap.

## 1.17.1
- Документация перестроена как читаемая иерархия по файлам: добавлен `docs/README.md` с картой чтения и распределением ответственности между `README`, roadmap, architecture, debugging, memory и release-документами.
- Снижено дублирование в релизной документации: `docs/RELEASES.md` теперь направляет к `RELEASE_CHECKLIST.md`, PR template и GitHub workflows вместо повторения тех же правил в нескольких местах.
- Уточнены `docs/ARCHITECTURE.md`, `docs/MEMORY.md` и `docs/DEBUGGING.md`: в них добавлены секции «как читать документ» и убраны устаревшие/повторяющиеся пункты, чтобы человеку было проще идти по документации последовательно.
- Обновлены `README.md`, `VERSION` и UI-баннер до версии `1.17.1`.

## 1.17.0
- Добавлен `docs/MEMORY.md` с планом миграции `PMM -> VMM -> kernel mapping policy`, чтобы следующий этап memory subsystem шёл по формальному контракту, а не через неявный identity-map baseline.
- Добавлены `docs/DEBUGGING.md` и `docs/RELEASES.md`: в репозитории зафиксированы типовые boot/runtime-сценарии диагностики, safe fallback-переключатели и релизный процесс `каждый PR = новая версия`.
- Добавлен `.github/pull_request_template.md`, а roadmap обновлён: закрыты пункты про VMM migration plan, debugging docs, releases docs и PR template.
- Обновлены `README.md`, `docs/ARCHITECTURE.md`, `VERSION` и UI-баннер до версии `1.17.0`.

## 1.16.0
- Модуль `timer` переведён с loop-based псевдотаймера на аппаратный PIT: канал 0 программируется в periodic rate generator mode, а ядро опрашивает latched-счётчик и детектирует прошедшие тики без зависимости от скорости основного цикла.
- В `kernel` heartbeat теперь инициализируется как аппаратный таймер `20 Hz`, поэтому UI-диагностика обновляется от реального источника времени, а не от числа итераций main loop.
- В roadmap закрыт соответствующий технический долг этапа C, README/architecture обновлены под новый таймерный контракт.
- Обновлены `VERSION`, UI-баннер и документация до `1.16.0`.

## 1.15.0

- В roadmap отмечены выполненными формализация иерархии модулей/порядка инициализации и отдельный архитектурный документ.

- В footer UI добавлена диагностика PMM (`PMM READY/WAIT`, total/free pages), а `kernel` публикует эти метрики через новый контракт `ui_set_memory_stats`.

## 1.13.0

## 1.12.3
- Исправлен критичный баг split в `kheap`: размер нового свободного блока теперь вычисляется без повторного добавления размера заголовка, что предотвращает выход за границы арены при последовательных аллокациях/слияниях.
- Усилена гарантия 16-байтного выравнивания: arena явно выровнена на 16 байт, а заголовок блока сделан кратным 16, поэтому указатели payload соответствуют `KHEAP_ALIGN`.
- Добавлены поясняющие комментарии в чувствительных местах allocator-кода.
- Обновлены `VERSION`, UI-баннер и документация до `1.12.3`.

## 1.12.2
- Исправлена ошибка в bootloader после релиза `1.12.1`: инкремент `current_lba` больше не читает 32 бита из 16-битного поля `sectors_this_read`.
- Теперь LBA обновляется корректно через `movzx` из `word`, что исключает повреждение следующего адреса чтения и повторяющийся `Disk error` в QEMU/BIOS.
- Обновлены `VERSION`, UI-баннер и документация до `1.12.2`.

## 1.12.1
- Исправлена загрузка крупных сборок с BIOS INT 13h Extensions: bootloader больше не пытается читать весь payload одним вызовом `AH=42h`.
- Добавлено поэтапное чтение kernel payload кусками до 127 секторов за операцию с корректным обновлением `LBA` и адреса назначения в памяти.
- Это устраняет сценарий `Booting from hard disk... Disk error` на части конфигураций QEMU/BIOS при росте `kernel.bin`.
- Обновлены `VERSION`, UI-баннер и документация до `1.12.1`.

## 1.12.0
- Добавлен базовый heap-аллокатор ядра (`kheap`) с фиксированной ареной, выравниванием по 16 байт, split/free и слиянием соседних свободных блоков.
- Для heap добавлены диагностические метрики `kheap_total_bytes`, `kheap_used_bytes`, `kheap_free_bytes`.
- Input ring buffer переведён на runtime-размещение через `kheap_alloc` с безопасным fallback на статический буфер, если heap недоступен на раннем этапе.
- В init-последовательность ядра добавлена инициализация `kheap` перед запуском input subsystem.
- Обновлены `VERSION`, UI-баннер и документация до `1.12.0`.

## 1.11.11
- Возвращены производительные дефолты сборки: `VIRTIO_GPU=1` и `HW_INTERRUPTS=1` по умолчанию.
- Точечно зафиксирована нестабильность IRQ-контура: в `idt_init` все линии PIC оставлены замаскированными в штатном runtime (мышь и heartbeat уже работают через polling), что устраняет спорадические IRQ как источник reboot-loop.
- В обработчике `IRQ1` добавлен безопасный drain PS/2 data-port только при `status&OBF`, чтобы спорадический IRQ не вычитывал пустой порт.
- Сохранён ручной контроль диагностики через флаги сборки `VIRTIO_GPU` и `HW_INTERRUPTS`.
- Обновлены `VERSION`, UI-баннер и документация до `1.11.11`.

## 1.11.10
- Точечно усилена стабильность IRQ-конфигурации PIC: в `idt_init` больше не размаскируется `IRQ2` (cascade) в штатном режиме, так как slave-IRQ сейчас не используются (`IRQ12` мыши остаётся polling).
- Это убирает потенциальный источник спорадических/пустых cascade IRQ на части VM-конфигураций, который мог повторно провоцировать нестабильность после `sti`.
- Фикс выравнивания стека в IRQ-stub и безопасный дефолт `HW_INTERRUPTS=0` сохранены; для диагностики IRQ-путь включается вручную (`HW_INTERRUPTS=1`).
- Обновлены `VERSION`, UI-баннер и документация до `1.11.10`.

## 1.11.9
- После регрессионной проверки `1.11.8` подтверждена нестабильность при включённых IRQ на части конфигураций: поэтому дефолтный профиль снова переведён в безопасный режим (`HW_INTERRUPTS=0`) до полного закрытия root-cause в IRQ/runtime-path.
- Фикс ABI-выравнивания стека в `idt_asm` из `1.11.8` сохранён; для целевой диагностики IRQ-путь по-прежнему доступен вручную через `make os.img HW_INTERRUPTS=1`.
- Таким образом штатная сборка остаётся стабильной (polling-путь), а рискованный сценарий изолирован за явным флагом.
- Обновлены `VERSION`, UI-баннер и документация до `1.11.9`.

## 1.11.8
- Найдена и исправлена вероятная корневая причина reboot-loop в IRQ-path: в `idt_asm` перед вызовом `idt_handle_irq` стек выравнивался неверно для SysV ABI (`rsp % 16`), что могло приводить к падениям внутри C-обработчика после `sti`.
- В IRQ-stub добавлена корректировка стека (`sub rsp, 8` после выравнивания), чтобы C-обработчик всегда стартовал с корректным 16-байтным выравниванием.
- После фикса аппаратные IRQ снова включены в стандартной сборке (`HW_INTERRUPTS=1` по умолчанию), а флаг `HW_INTERRUPTS` оставлен для диагностики.
- Обновлены `VERSION`, UI-баннер и документация до `1.11.8`.

## 1.11.7
- Добавлен ещё один безопасный режим по умолчанию для изоляции reboot-loop: аппаратные прерывания (`sti`) теперь выключены в стандартной сборке (`HW_INTERRUPTS=0`), так как текущий runtime уже использует polling-путь для мыши и программный heartbeat-таймер.
- В `Makefile` добавлен флаг `HW_INTERRUPTS` (по умолчанию `0`), который пробрасывается в `kernel` как `WOOS_ENABLE_HW_INTERRUPTS`; для диагностики IRQ-пути его можно включить вручную через `make os.img HW_INTERRUPTS=1`.
- Это исключает весь класс сбоев после `sti` (IRQ storm / fault в IRQ-path) из штатного boot и оставляет управляемый возврат к IRQ-режиму только для целевой проверки.
- Обновлены `VERSION`, UI-баннер и документация до `1.11.7`.

## 1.11.6
- Введён «безопасный режим» рендера по умолчанию: `virtio_gpu_renderer` теперь отключён в стандартной сборке (`WOOS_ENABLE_VIRTIO_GPU=0`), чтобы исключить нестабильный MMIO/virtqueue-path как источник циклического reboot и глитчей.
- В `Makefile` добавлен явный флаг `VIRTIO_GPU` (по умолчанию `0`): для целевой диагностики драйвер можно включить вручную через `make os.img VIRTIO_GPU=1`.
- Сохранён текущий код `virtio-gpu` (включая защиту BAR/MMIO), но его активация теперь контролируемая, чтобы сначала подтвердить стабильность базового framebuffer-path.
- Обновлены `VERSION`, UI-баннер и документация до `1.11.6`.

## 1.11.5
- Исправлена ещё одна вероятная причина циклического reboot/мигания экрана на конфигурациях с `virtio-vga`: парсинг MMIO BAR в `virtio_gpu_renderer` теперь корректно обрабатывает 64-битные BAR (`BAR0+BAR1`), а не только 32-битный нижний регистр.
- Добавлен защитный фильтр адресов capability-регионов virtio: если MMIO-регион расположен выше 4 ГиБ (вне текущего identity-map stage2), драйвер не трогает этот регион и остаётся на software framebuffer fallback вместо `#PF -> reset`.
- Обновлены `VERSION`, UI-баннер и документация до `1.11.5`.

## 1.11.4
- Устранена ещё одна потенциальная причина циклической перезагрузки после `sti`: теперь в IDT установлены отдельные IRQ-stub'ы на весь диапазон PIC-векторов `32..47`, а не только на `IRQ1/IRQ12`.
- Это гарантирует корректный `EOI` даже для неожиданных/spurious IRQ (например, `IRQ2/IRQ7/IRQ15`) и убирает зависимость стабильности от того, какие линии в реальности дёргает эмулятор/чипсет.
- Для исключений с аппаратным `error code` добавлены также векторы `#VC` (`29`) и `#SX` (`30`) в `idt_stub_ignore_errcode`, чтобы не допускать повреждения стека возврата на более строгих CPU/VM-конфигурациях.
- Обновлены `VERSION`, UI-баннер и документация до версии `1.11.4`.

## 1.11.3
- Исправлена потенциальная причина нестабильного boot после изменений `1.11.0`: в `idt_init` IRQ12 (мышь) снова оставлен замаскированным, так как текущий `mouse`-драйвер работает в polling-режиме и не обрабатывает поток байтов в IRQ-контексте.
- Это устраняет конфликт «IRQ-driven mouse + polling mouse», из-за которого в некоторых VM/конфигурациях контроллер мог генерировать шквал прерываний и провоцировать зависание/циклическую перезагрузку после `sti`.
- В `idt_handle_irq` добавлен безопасный drain PS/2 data-port для ветки `IRQ12` на будущее (если IRQ12 будет снова размаскирован).
- Обновлены `VERSION` и UI-баннер до версии `1.11.3`.

## 1.11.2
- Найдена и исправлена реальная причина циклической перезагрузки: в IDT все векторы по умолчанию указывали на `idt_stub_ignore` с `iretq`, но для исключений с аппаратным `error code` (например, `#GP`, `#PF`, `#DF`) это ломало стек возврата.
- В `idt_asm.asm` добавлен отдельный stub `idt_stub_ignore_errcode`, который снимает `error code` (`add rsp, 8`) перед `iretq`.
- В `idt.c` векторы исключений с `error code` переведены на новый stub, чтобы исключить каскад `#GP/#DF` и последующий triple fault/reset.
- Обновлены `VERSION` и UI-баннер до версии `1.11.2`.

## 1.11.1
- Исправлен критический баг обработки аппаратных IRQ в `idt_asm`: перед вызовом `idt_handle_irq` добавлено явное выравнивание стека по SysV ABI и восстановление исходного `rsp` перед `iretq`.
- Это устраняет нестабильный `#GP -> double fault -> triple fault` при включении прерываний (`sti`) и предотвращает циклический reset CPU на запуске в QEMU/KVM.
- Обновлены `VERSION` и UI-баннер до версии `1.11.1`.

## 1.11.0

- Добавлены счётчики IRQ (`idt_keyboard_irq_count`, `idt_mouse_irq_count`) и интеграция в UI-footer как диагностическая строка `IRQ K:.... M:....`.
- В `kernel` включены аппаратные прерывания (`sti`) после инициализации устройств, а heartbeat-тик обновляет UI-диагностику по IRQ.
- Обновлены `README.md`, `DEVELOPMENT_PLAN.md`, `VERSION` и UI-баннер до версии `1.11.0`.

## 1.10.1

## 1.10.0
- В `drivers/virtio_gpu_renderer` добавлен command-oriented рендер-пайплайн: `fb`/UI теперь отправляют draw-команды (`fill/rect/glyph`) в renderer, а не пишут пиксели напрямую в stage2 framebuffer при активном `virtio-gpu`.
- Добавлена отдельная draw-surface в RAM для GPU-пути: ресурс `virtio-gpu` получает backing на этой поверхности, после чего dirty-области публикуются через virtqueue (`TRANSFER_TO_HOST_2D` + `RESOURCE_FLUSH`).
- Добавлена negotiated-поддержка `VIRTIO_GPU_F_VIRGL` (virgl capability bit) с безопасным fallback, если функция недоступна на устройстве.
- Сохранены текущие init-flow (`early -> platform -> drivers -> ui`), driver-layer разделение и существующий `fb/ui` API; software framebuffer fallback остаётся рабочим для non-virtio/legacy-path.

## 1.9.0
- Добавлен модуль `drivers/virtio_gpu_renderer`, который в driver-stage определяет `virtio-gpu` по PCI и при валидном MMIO BAR переключает `video->framebuffer` на framebuffer устройства.
- Сохранён текущий init-flow (`early -> platform -> drivers -> ui`): инициализация рендера virtio встроена в `INIT_DRIVERS` без изменений API UI/fb.
- Добавлен безопасный fallback: если `virtio-gpu` не найден или BAR невалиден, ядро продолжает рендерить в framebuffer, переданный `stage2`.
- В `fb_present_rect` добавлен backend-hook `virtio_gpu_renderer_present_rect(...)` как фундамент для следующего шага (virtqueue `TRANSFER_TO_HOST_2D` / `RESOURCE_FLUSH`) без поломки dirty-rect/double-buffer pipeline.
- Обновлены документация и UI-баннер до версии `1.9.0`.
