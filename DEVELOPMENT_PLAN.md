# WoOS Development Plan

> Цель: эволюционно построить инженерно аккуратную x86_64 ОС с чистой архитектурой, предсказуемым графическим стеком и релизным процессом, пригодным для командной разработки.

## 1) Инженерные принципы проекта

### Архитектурные
- **Слои и ответственность:** bootloader → kernel core → subsystems (`mm`, `irq`, `video`, `input`, `fs`) → UI.
- **Явные контракты:** все интерфейсы между подсистемами описаны в заголовках и коротких design notes.
- **Минимальный freestanding baseline:** не тащить runtime-зависимости без необходимости.

### Процессные
- **Каждый PR = новая версия** (обновление `VERSION`, `CHANGELOG`, docs).
- **Малые инкременты:** изменения с проверяемым результатом за один PR.
- **Документация как артефакт разработки:** архитектурные решения фиксируются в repo, а не в обсуждениях.

### Качество
- Единые предупреждения компилятора (`-Wall -Wextra`) без игнорирования критичных сигналов.
- Контроль регрессий через CI-сборку `os.img` и проверку сигнатур/маркеров.
- Релизы собираются автоматически и воспроизводимо из GitHub Actions.

## 3) Подробный roadmap

## Этап A — Platform Foundation (ядро платформы)
**Цель:** сделать запуск и графику устойчивыми и предсказуемыми.

### A.1 Boot & Kernel Bootstrapping
- [x] Ввести документированную последовательность init-стадий (`early`, `platform`, `drivers`, `ui`).
- [x] Зафиксировать ABI между `stage2` и `kernel` (структура boot info, версия контракта).
- [x] Добавить sanity-check структуры boot info на стороне ядра.

### A.2 Графический pipeline
- [x] Перейти к частичной перерисовке (dirty rectangles).
- [x] Выделить двойную буферизацию как опциональный режим (compile-time флаг).
- [x] Добавить профилировочные метки перерисовки (число dirty-областей / кадр).

**Definition of Done (DoD) этапа A:**
- Стабильная сборка и запуск `os.img`.
- UI перерисовывается без full-screen flicker при локальных изменениях.

---

## Этап B — Input & Event System
**Цель:** управляемый интерактивный UI.

### B.1 PS/2 и курсор
- [x] Инициализация PS/2-контроллера и чтение пакетов мыши.
- [x] Поддержка состояния курсора (`x/y`, `buttons`, `clamp to screen`).
- [x] Отрисовка курсора без полного redraw (save/restore фонового блока).

### B.2 Event Dispatcher
- [x] Очередь событий фиксированного размера (ring buffer).
- [x] Нормализация событий (mouse move, click, key press, timer tick).
- [x] UI-handlers для hover/click на панели и тестовом окне.

**DoD этапа B:**
- Курсор двигается плавно.
- Клики обрабатываются детерминированно, без «залипания» состояния.

---

## Этап C — Memory & Interrupts
**Цель:** сформировать системный каркас для будущих сервисов ядра.

### C.1 Interrupt Subsystem
- [x] IDT skeleton с базовым default ISR-stub (этап 1 для подсистемы прерываний).
- [x] Базовый heartbeat-таймер (polling mode) с интеграцией в event loop и UI-диагностику.
- [x] Базовый `virtio-gpu` renderer-path: PCI-probing + переключение active framebuffer на BAR с fallback на stage2 framebuffer.
- [x] Обвязка IRQ для клавиатуры/мыши.

### C.2 Memory Subsystem
- [x] PMM (stack-based allocator для физических страниц на основе BIOS E820 memory map).
- [x] Базовый heap allocator ядра (рантайм-аллокатор для внутренних структур).
- [x] План миграции к VMM (карта памяти, page tables, kernel mapping policy).

**DoD этапа C:**
- Прерывания стабильны, panic path информативен.
- Память выделяется/освобождается под внутренние структуры без порчи состояния.

---

## Этап D — Storage & VFS
**Цель:** чтение данных с диска через единый API.

### D.1 Дисковый минимум
- [x] Транспортный слой блочного чтения (LBA read abstraction).
- [x] Тестовый интерфейс чтения фиксированных секторов.

### D.2 VFS skeleton
- [x] Общий контракт `open/read/close/readdir`.
- [x] Подключение первой минимальной ФС (read-only сценарий).
- [x] Каркас прав доступа (ACL-like модель, упрощённая).

**DoD этапа D:**
- Ядро читает файл из образа через VFS-API.

---

## Этап E — UX, Desktop и расширяемость
**Цель:** удобный и масштабируемый интерфейс.

- [x] Компоновка UI через структуры layout (панель, контент, статус-строка).
- [ ] Темы/палитры и конфигурация визуального профиля.
- [ ] Примитивный window manager (z-order, фокус, drag в упрощённом виде).

**DoD этапа E:**
- У пользователя есть предсказуемая модель взаимодействия и читаемый UI-статус.

## 4) Release Engineering и DevEx

### Автоматизация релизов
- [x] Build workflow собирает `os.img` и публикует artifact.
- [x] Release workflow после merge одобренного PR автоматически:
  - [x] валидирует наличие `APPROVED` review,
  - [x] собирает `os.img` из merge-коммита,
  - [x] формирует notes из `CHANGELOG.md` для версии из `VERSION`,
  - [x] публикует GitHub Release с тегом `vX.Y.Z` и приложенным образом.

### Стандарты PR
- [x] PR template с чеклистом (версия, changelog, docs, тесты).
- [ ] Политика именования веток и Conventional-style заголовки PR.
- [ ] Явное описание риска и rollback-стратегии в сложных PR.

### Документация
- [x] Добавить `docs/ARCHITECTURE.md` (слои, init flow, контракты).
- [x] Добавить `docs/DEBUGGING.md` (типовые проблемы boot/kernel).
- [x] Добавить `docs/RELEASES.md` (как выпускать теги и что генерируется автоматически).

## 5) Метрики прогресса
- **Стабильность:** % успешных CI-сборок за 30 дней.
- **Скорость итерации:** медиана времени от PR до merge.
- **Качество релизов:** доля релизов без hotfix-коммитов в первые 48 часов.
- **Прозрачность:** все значимые изменения покрыты `CHANGELOG` и docs.
