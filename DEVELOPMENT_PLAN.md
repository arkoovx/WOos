# WoOS Development Plan

> Статус на 2026-03-28: исходный план (этапы A–E) **завершён** и переведён в архивный режим.

## 1) Пройденный этап (сильно сокращённый архив)

Базовый roadmap закрыт:
- [x] Boot + long mode + стабильный вход в ядро.
- [x] Формализованный init-flow `early -> platform -> drivers -> ui`.
- [x] Базовые подсистемы `mm/irq/video/input/storage/vfs`.
- [x] Рабочий графический UI-стек и runtime-диагностика.
- [x] CI/release-контур и правило `каждый PR = новая версия`.

Это считается завершённым фундаментом WoOS и больше не ведётся как отдельный детализированный roadmap.

---

## 2) Новый вектор: интеграция проверенных модулей

WoOS дальше развивается по модели **"своё ядро + готовые проверенные кирпичики"**:
- Архитектура, порядок инициализации, memory model, event loop и UI-стек остаются зоной ответственности WoOS.
- Низкоуровневые инфраструктурные блоки (сетевой стек, часть драйверов, FS-реализации) предпочтительно подключаются как внешние модули, а не пишутся заново с нуля.

Ключевая цель: меньше времени на переизобретение стандартизированных подсистем и больше времени на интеграцию, устойчивость и архитектурную целостность.

---

## 3) Контур интеграции (как добавляем внешние компоненты)

1. **Источник**: берём зрелую реализацию с ясной лицензией (предпочтительно MIT/BSD/совместимые).
2. **Изоляция**: кладём код во внешний каталог (рекомендуется `external/<module>`).
3. **Адаптер WoOS**: пишем тонкий слой `drivers/<module>_adapter` или `net/<module>_port`, который:
   - связывает внешний модуль с API WoOS,
   - не протаскивает детали модуля в UI и core,
   - поддерживает safe fallback, если модуль недоступен.
4. **Инициализация**: подключаем модуль в stage `drivers` без размывания init-flow.
5. **Диагностика**: добавляем минимальные runtime-метрики и статус в UI/логике диагностики.
6. **Документация**: фиксируем контракт интеграции в `docs/ARCHITECTURE.md` и changelog.

---

## 4) Приоритеты следующего цикла

### P1. Сеть (рекомендуемый первый шаг)
- Подготовить место под внешний стек (`external/lwip` как целевой формат).
- Добавить план порта и API-границу `driver -> lwIP adapter -> WoOS network API`.
- Первый целевой результат: базовая инициализация стека + диагностический статус в UI.

### P2. Драйверный слой
- Продолжать курс на стандартные протоколы (как в текущем `virtio_gpu`).
- Для storage-path держать курс на замену узких мест ATA PIO на более зрелый путь (AHCI/NVMe) через готовые реализации.

### P3. ФС-эволюция
- Оставить `WOFS` как минимальный baseline.
- Для практического использования планировать порт готовой FAT32/exFAT-реализации в модульном стиле.

---

## 5) Критерии качества нового подхода

- Новые внешние модули не ломают контракты init-flow.
- Любой модуль можно отключить флагом сборки и получить предсказуемый fallback.
- Архитектурная ответственность остаётся в WoOS core, а не "уезжает" внутрь внешней библиотеки.
- Каждый интеграционный PR обновляет `VERSION`, `CHANGELOG` и профильную документацию.
