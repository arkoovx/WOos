# WoOS Architecture

Этот документ фиксирует текущую архитектуру WoOS, порядок инициализации и контракты между ключевыми подсистемами.
Его цель — сделать развитие ядра предсказуемым: новые подсистемы должны встраиваться в уже описанные слои, а не обходить их.

## Как читать этот документ

- Если вы только входите в проект, начните с `README.md` и `DEVELOPMENT_PLAN.md`.
- Этот файл является архитектурным центром: он описывает слои, init-flow и инварианты.
- После него переходите в специализированные документы: `docs/MEMORY.md` для memory stack, `docs/DEBUGGING.md` для практической диагностики и `docs/RELEASES.md` для process/release-контура.

## 1. Архитектурные слои

WoOS развивается по слоистой модели:

1. **Bootloader** — `boot.asm`, `stage2.asm`.
   - Загружает ядро с диска.
   - Переключает CPU в long mode.
   - Подготавливает и передаёт `boot_info` в ядро.
2. **Kernel Core** — `kernel.c`, `kernel.h`.
   - Определяет init-flow.
   - Оркестрирует запуск платформенных и runtime-подсистем.
   - Содержит главный цикл обработки событий.
3. **Platform / Hardware Abstraction** — `fb`, `idt`, `timer`, `mouse`, `pci`, `pmm`, `storage`, `vfs`.
   - Даёт ядру минимально безопасный доступ к видео, IRQ, аппаратному PIT-таймеру, input и физической памяти.
4. **Drivers** — `drivers/virtio_gpu_renderer` и будущие драйверы устройств.
   - Расширяют базовые платформенные сервисы.
   - Могут ускорять существующие пути, но обязаны иметь безопасный fallback.
5. **Runtime Services** — `kheap`, `input`.
   - Предоставляют внутренние сервисы ядра: динамическую память и очередь событий.
6. **UI Layer** — `ui`.
   - Использует framebuffer API и события input.
   - Не должен напрямую управлять железом или boot-параметрами.

## 2. Init-flow ядра

Порядок инициализации является частью архитектурного контракта.

### Stage `early`
Файл: `kernel.c`

Назначение:
- sanity-check и нормализация `boot_info`;
- подготовка безопасных значений по умолчанию для framebuffer и версии boot ABI.

Правило:
- на этом этапе нельзя зависеть от heap, UI, input-очереди или драйверов.

### Stage `platform`
Файлы: `fb`, `idt`

Назначение:
- активация базового framebuffer-path через `fb_init`;
- загрузка IDT через `idt_init`.

Правило:
- платформенные подсистемы должны быть готовы раньше, чем начнутся драйверы и UI.
- код этого этапа не должен зависеть от `ui`.

### Stage `drivers`
Файлы: `drivers/virtio_gpu_renderer`, `pmm`, `kheap`, `input`, `timer`, `storage`, `vfs`

Назначение:
- инициализация ускоренного видео-path (`virtio-gpu`) с fallback на software framebuffer;
- запуск PMM как базы для будущего VMM;
- запуск heap allocator;
- инициализация input queue;
- запуск heartbeat-таймера;
- инициализация storage transport и self-check чтения фиксированного сектора;
- инициализация VFS skeleton с унифицированным read-only API (`open/read/close/readdir`) и базовым ACL-like permission scaffold (owner/group/other).

Правила:
- все runtime-структуры, требующие динамической памяти, должны подниматься только после `kheap_init`;
- драйверы обязаны корректно деградировать в fallback-режим;
- новые драйверы должны подключаться именно на этой стадии, если им не нужен userspace/UI.

### Stage `ui`
Файл: `ui.c`

Назначение:
- первая полная отрисовка desktop UI;
- публикация начального состояния диагностик.

Правила:
- UI работает только поверх уже подготовленных сервисов `fb` и `input`;
- UI не должен менять порядок init нижележащих подсистем.

### Runtime loop
После init ядро переходит в основной цикл:
- короткая пауза через `pause` для снижения бессмысленной загрузки CPU;
- polling heartbeat через `timer_poll_tick` с аппаратным PIT channel 0;
- polling мыши через `mouse_poll`;
- drain очереди `input_pop`;
- публикация runtime-диагностики в UI;
- частичный redraw через `ui_render_dirty`.

## 3. Контракты между подсистемами

### 3.1 Bootloader → Kernel
Контракт задаётся структурой `video_info_t` в `kernel.h`.

Обязательные поля:
- framebuffer-адрес и геометрия;
- `magic`, `version`, `size`;
- при ABI v2: карта памяти BIOS E820.

Правила:
- ядро не доверяет входной структуре без проверки;
- `sanitize_boot_info` обязан сохранить boot даже при частично некорректных данных;
- расширение ABI допустимо только через версионирование, без поломки старых путей.

### 3.2 Kernel Core → Platform / Drivers
`kernel.c` управляет только порядком вызовов и не должен дублировать внутреннюю логику подсистем.

Правила:
- каждый модуль предоставляет узкий init/API-контракт через заголовок;
- глобальные побочные эффекты должны быть локализованы внутри соответствующего модуля;
- если модулю нужен fallback, он должен быть инкапсулирован внутри модуля, а не в `kernel.c`.

### 3.3 Platform / Drivers → UI
UI получает только абстракции:
- framebuffer API из `fb.h`;
- input events из `input.h`;
- runtime-метрики через явные setter-функции `ui_set_*`.

Правила:
- UI не должен читать MMIO/PCI/PS2-регистры напрямую;
- UI не должен знать детали boot ABI, кроме уже нормализованных полей `video_info_t`.

### 3.4 Memory stack
Текущая модель памяти:
- `pmm` управляет физическими страницами 4 КиБ;
- `kheap` даёт runtime-аллокатор для внутренних структур;
- VMM пока не реализован, поэтому модули обязаны учитывать ограничения текущего identity-mapped окружения.

Правила:
- PMM не зависит от heap;
- heap не должен требовать VMM для своей базовой работы;
- код, работающий с MMIO/BAR, обязан учитывать, что адрес выше 4 ГиБ сейчас может быть недоступен.

## 5. Инварианты проекта

Следующие правила считаются обязательными для новых изменений:

1. **Safe fallback важнее ускорения.**
   - Любой ускоренный или аппаратно-зависимый путь должен сохранять рабочий fallback.
2. **Init-flow не размывается.**
   - Новые подсистемы добавляются в существующие стадии, а не через хаотичные вызовы из разных мест.
3. **UI не владеет железом.**
   - UI работает только через уже нормализованные API.
4. **Boot ABI меняется только версионированно.**
   - Нельзя тихо менять layout boot-структур.
5. **Диагностика — часть архитектуры.**
   - Если новая подсистема важна для boot/runtime, у неё должен быть хотя бы минимальный диагностический сигнал.

## 6. Ограничения текущего baseline

На момент версии `1.19.0` архитектура всё ещё имеет сознательные ограничения:
- heartbeat уже использует аппаратный PIT, но пока остаётся polling-based, а не IRQ-driven;
- полноценный VMM отсутствует;
- VFS пока минимальный и read-only (фиксированный каталог + тестовый файл), ACL-каркас упрощённый (без userspace/login), полноценная файловая система поверх него ещё не реализована;
- UI уже поддерживает примитивный window manager (z-order/focus/drag), но пока остаётся встроенным в `ui.c` без модульного API для приложений.

Эти ограничения допустимы, пока новые изменения не нарушают описанные выше контракты.

## 7. Ближайшее развитие архитектуры

Следующий цикл WoOS строится по принципу **"core-owned architecture, module-based implementation"**.

Приоритетные направления:
- интеграция внешних зрелых подсистем через тонкие адаптеры WoOS (в первую очередь сетевой стек);
- сохранение текущих архитектурных границ: `kernel.c` владеет только orchestration, UI остаётся поверх абстракций, а железо инкапсулируется в platform/drivers;
- обязательный safe fallback для каждого внешнего модуля и флаги сборки для управляемой деградации.

Практическое правило для новых интеграций:
- внешний код размещается изолированно (рекомендуемо: `external/<module>`),
- WoOS-специфичная логика живёт в адаптере (`drivers/*_adapter` или аналог),
- подключение проходит строго через stage `drivers`, без обхода init-flow.
