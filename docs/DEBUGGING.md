# WoOS Debugging Guide

Этот документ собирает типовые сценарии отладки WoOS на текущем baseline.
Он нужен как практическое дополнение к архитектуре: где искать проблему, какие симптомы о чём говорят и какие безопасные переключатели уже есть в репозитории.

## Как читать этот документ

- Сначала откройте `docs/ARCHITECTURE.md`, если вам нужно понять ожидаемый init-flow и межмодульные контракты.
- Затем используйте этот файл как практическую памятку: от симптома к проверкам и безопасным fallback-режимам.
- Если проблема относится именно к памяти, после базовой локализации переходите в `docs/MEMORY.md`.


## 1. Базовая стратегия диагностики

Если после изменения система перестала вести себя предсказуемо, используйте такой порядок:

1. Проверить, собирается ли `os.img` без предупреждений/ошибок компоновки.
2. Запустить систему в безопасной конфигурации без подозрительного ускоренного пути.
3. Сравнить поведение safe fallback и стандартной конфигурации.
4. Проверить, на какой стадии init-flow происходит деградация:
   - `early`;
   - `platform`;
   - `drivers`;
   - `ui`;
   - runtime loop.
5. Только после этого углубляться в конкретный модуль.

## 2. Полезные команды

Базовая сборка:
```bash
make clean
make os.img
```

Проверка layout образа:
```bash
make verify-layout
```

Безопасный framebuffer fallback без `virtio-gpu`:
```bash
make clean
make os.img VIRTIO_GPU=0
```

Диагностический режим без аппаратных IRQ:
```bash
make clean
make os.img HW_INTERRUPTS=0
```

Диагностика без double buffer:
```bash
make clean
make os.img DBL_BUFFER=0
```

## 3. Типовые симптомы и что они обычно значат

### Симптом: `Disk error` сразу после `Booting from hard disk...`
Чаще всего это означает проблему в boot-path чтения payload с диска:
- повреждён образ `os.img`;
- регрессия в логике чтения секторов в `boot.asm`/`stage2.asm`;
- kernel payload вырос и снова затронул чувствительный участок загрузки.

Проверки:
- пересобрать образ с `make clean && make os.img`;
- запустить `make verify-layout`;
- посмотреть последние изменения в `boot.asm`, `stage2.asm`, `linker.ld`.

### Симптом: циклическая перезагрузка или мгновенный reset
Обычно это означает одно из двух:
- fault в раннем boot/runtime-path;
- некорректная работа с IRQ/IDT/стеком обработчика.

Проверки:
- собрать и запустить с `HW_INTERRUPTS=0`;
- проверить изменения в `idt.c`, `idt_asm.asm`, `kernel.c`;
- отдельно убедиться, что новый код не трогает MMIO-адреса вне текущего доступного диапазона.

### Симптом: чёрный экран или UI не появляется
Вероятные причины:
- проблема инициализации framebuffer;
- сломан fallback между `virtio-gpu` и software framebuffer;
- ранний сбой до этапа `ui`.

Проверки:
- собрать с `VIRTIO_GPU=0`;
- проверить изменения в `fb.c`, `drivers/virtio_gpu_renderer`, `kernel.c`;
- убедиться, что `sanitize_boot_info` всё ещё сохраняет корректные fallback-значения.

### Симптом: курсор/мышь не реагируют
Вероятные причины:
- регрессия в `mouse` polling-path;
- ошибка в очереди `input`;
- UI получает события, но не обновляет dirty-region.

Проверки:
- проверить изменения в `mouse.c`, `input.c`, `ui.c`;
- убедиться, что основной цикл ядра продолжает вызывать `mouse_poll()` и `ui_render_dirty()`.

### Симптом: footer/overlay обновляется слишком редко или «рывками»
Вероятные причины:
- изменена пауза основного цикла;
- ошибка в heartbeat-логике `timer_poll_tick()`;
- неожиданный рост стоимости рендера dirty-region.

Проверки:
- проверить `kernel.c`, `timer.c`, `ui.c`;
- сравнить поведение со стандартной сборкой и конфигурацией `DBL_BUFFER=0`.

## 4. Модульные зоны риска

### Bootloader (`boot.asm`, `stage2.asm`)
Особенно чувствителен к:
- изменению layout/kernel size;
- неверной арифметике LBA/секторов;
- поломке boot ABI.

Правило:
- любые изменения здесь должны сопровождаться повторной проверкой `make verify-layout`.

### IDT / IRQ (`idt.c`, `idt_asm.asm`)
Особенно чувствительны к:
- выравниванию стека;
- правильной отправке `EOI`;
- смешению polling и IRQ-driven путей.

Правило:
- если нет уверенности в IRQ-path, сначала подтверждайте стабильность через `HW_INTERRUPTS=0`.

### Memory (`pmm`, `kheap`)
Особенно чувствительна к:
- выравниванию;
- повторному освобождению;
- предположениям о доступности адресов выше текущего baseline.

Правило:
- не вводить неявную зависимость от будущего VMM до появления формального memory API.

### Video / virtio-gpu
Особенно чувствительны к:
- MMIO/BAR-адресам;
- частичной перерисовке;
- согласованности software и accelerated path.

Правило:
- ускоренный путь всегда должен иметь воспроизводимый fallback.

## 5. Как локализовать регрессию

Минимальный безопасный алгоритм:
1. Подтвердить, что `make os.img` проходит.
2. Проверить safe fallback (`VIRTIO_GPU=0`, при необходимости `HW_INTERRUPTS=0`).
3. Сузить проблему до init-stage или runtime.
4. Проверить только затронутые модули, а не весь стек сразу.
5. Обновить документацию, если найден новый класс сбоя или новый диагностический режим.

## 6. Когда нужно обновлять эту памятку

Обновляйте `docs/DEBUGGING.md`, если в репозитории появляется:
- новый безопасный build/runtime-флаг;
- новый типовой сценарий сбоя;
- новая обязательная проверка для boot/kernel path;
- новая подсистема, у которой есть собственный характерный failure mode.
