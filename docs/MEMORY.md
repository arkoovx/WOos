# WoOS Memory Roadmap

Этот документ фиксирует план эволюции memory stack WoOS: от текущего PMM к минимальному VMM и далее к предсказуемой политике kernel mappings.

Цель документа — закрыть архитектурный пробел между уже реализованным `pmm` и будущими подсистемами, которым понадобятся виртуальные отображения, guard-страницы, управляемый доступ к MMIO и более строгая модель адресного пространства.

## Как читать этот документ

- Сначала откройте `docs/ARCHITECTURE.md`, чтобы увидеть общие ограничения текущего baseline и место memory stack в init-flow.
- Затем используйте этот файл как специализированный roadmap по памяти: он не повторяет полную архитектуру, а уточняет только memory-related решения.
- Если вы расследуете memory-регрессию, после чтения этого документа переходите в `docs/DEBUGGING.md`.


## 1. Текущий baseline

На версии `1.17.1` в WoOS уже есть:
- BIOS E820 memory map, передаваемая из `stage2` в ядро;
- `pmm` как stack-based allocator физических страниц 4 КиБ;
- `kheap` как рантайм-аллокатор для внутренних структур ядра;
- identity-mapped окружение раннего boot, на которое пока опираются все модули.

Текущие ограничения:
- отдельный VMM ещё не реализован;
- код не может безопасно полагаться на ленивое отображение новых диапазонов;
- MMIO/BAR выше 4 ГиБ нельзя считать доступными автоматически;
- heap живёт в заранее доступной памяти и пока не умеет расширяться страницами из PMM.

## 2. Цели миграции

Переход к VMM должен решить четыре задачи:

1. **Явная карта адресного пространства ядра.**
   Чтобы новые модули не зависели от случайного identity-map baseline.
2. **Контролируемые page-table операции.**
   Чтобы ядро умело отображать RAM и MMIO по предсказуемым правилам.
3. **Основа для роста heap и будущих runtime-сервисов.**
   Чтобы `kheap` можно было расширять за счёт страниц из PMM.
4. **Безопасная база для storage, drivers и user/kernel-границ в будущем.**
   Чтобы развитие подсистем шло на устойчивой memory model, а не на boot-хаке.

## 3. Принципы memory stack

### 3.1 PMM и VMM остаются раздельными
- `pmm` отвечает только за физические страницы.
- `vmm` отвечает только за виртуальные отображения.
- Ни один модуль не должен смешивать эти две ответственности в одном API.

### 3.2 Safe fallback важнее полноты
- Сначала внедряется минимальный, но диагностируемый VMM для ядра.
- Только после этого поверх него расширяются heap, MMIO mapping и новые драйверы.

### 3.3 Boot identity map считается временным слоем
- Любая новая подсистема должна по возможности использовать явные VMM-контракты.
- Прямая зависимость от identity mapping допустима только в раннем boot или в явно задокументированном legacy-path.

### 3.4 Kernel mappings должны быть предсказуемыми
- Один и тот же тип памяти должен отображаться по одинаковым правилам.
- Политика атрибутов страниц должна быть централизована, а не размазана по драйверам.

## 4. Целевая поэтапная миграция

## Этап M1 — Memory Map Policy

**Цель:** формально описать, какие диапазоны адресов использует ядро.

Что вводим:
- фиксированные виртуальные области для:
  - образа ядра;
  - heap;
  - временных page-table/служебных отображений;
  - MMIO windows;
  - будущего direct-map региона физической памяти.
- константы и design notes для этих диапазонов.

DoD:
- карта адресного пространства описана в repo и не зависит от устных договорённостей;
- новые memory-related изменения опираются на эту карту.

## Этап M2 — Minimal VMM Core

**Цель:** добавить минимальный модуль `vmm`, который умеет отображать и снимать отображения страниц ядра.

Минимальный API:
- `vmm_init(...)`;
- `vmm_map_page(virt, phys, flags)`;
- `vmm_unmap_page(virt)`;
- `vmm_translate(virt)` для диагностик;
- `vmm_is_ready()`.

Требования:
- работа только в kernel-space;
- размер страницы — 4 КиБ как базовый путь;
- явные флаги `present/writable/no-exec` с возможностью расширения;
- безопасный отказ при попытке работать вне допустимых диапазонов.

DoD:
- ядро умеет создать и поддерживать собственные page-table mappings;
- есть минимальная диагностика состояния VMM.

## Этап M3 — Kernel Mapping Policy

**Цель:** зафиксировать правила отображения разных типов памяти.

Политика по типам:
- **Kernel text/rodata** — по возможности read-only, `NX` для данных выключен только там, где нужен исполняемый код.
- **Kernel data/bss/heap** — writable, non-executable.
- **Framebuffer/MMIO** — writable, с отдельным helper API и явной проверкой диапазонов.
- **Служебные таблицы/буферы драйверов** — writable, без исполняемых прав.

DoD:
- правила отображения оформлены как единый контракт;
- драйверы не выставляют page flags произвольно.

## Этап M4 — Heap on PMM+VMM

**Цель:** отвязать рост heap от фиксированной стартовой арены.

Что меняется:
- `kheap` получает возможность запрашивать новые страницы у `pmm`;
- новые физические страницы отображаются через `vmm` в выделенный heap-диапазон;
- fallback на стартовую арену сохраняется на раннем этапе миграции.

DoD:
- heap можно расширять без изменения bootloader;
- OOM-сценарии диагностируются явно.

## Этап M5 — MMIO Mapping Helpers

**Цель:** убрать из драйверов прямые предположения о доступности BAR-адресов.

Что вводим:
- helper для временного/постоянного отображения MMIO-регионов;
- валидация alignment/size/overlap;
- единая точка для отказа, если регион нельзя безопасно отобразить.

DoD:
- драйверы работают через memory API, а не через «надеюсь, этот BAR уже доступен».

## 5. Предлагаемая структура модулей

Минимальная целевая структура:
- `pmm.c/.h` — физические страницы;
- `vmm.c/.h` — page tables и map/unmap API;
- `kheap.c/.h` — allocator поверх уже доступной виртуальной области;
- `docs/MEMORY.md` — policy и roadmap;
- `docs/DEBUGGING.md` — типовые сценарии диагностики memory/boot path.

## 6. Риски и ограничения миграции

### Риск 1: сломать текущий стабильный boot
Смягчение:
- внедрять VMM инкрементально;
- сохранять ранний identity-map path до завершения M2/M3;
- не трогать boot ABI без отдельной необходимости.

### Риск 2: потерять доступ к framebuffer/MMIO
Смягчение:
- сначала описать mapping policy;
- только потом переводить драйверы на новые helper-функции.

### Риск 3: смешать обязанности PMM и VMM
Смягчение:
- API должны возвращать либо физические страницы, либо виртуальные mapping-операции, но не оба смысла сразу.

## 7. Практический порядок ближайших PR

Рекомендуемая последовательность после этого документа:
1. Подготовить константы и комментарии для будущей kernel virtual memory map.
2. Добавить минимальный `vmm` без миграции heap.
3. Перевести `kheap` на опциональное расширение через PMM+VMM.
4. Добавить helper для MMIO mappings и затем адаптировать `virtio_gpu_renderer`.

## 8. Что не входит в этот этап

Этот документ **не** вводит:
- userspace address spaces;
- copy-on-write;
- demand paging;
- swapping;
- полноценную изоляцию процессов.

Сейчас задача гораздо уже: перевести ядро с неявной boot-memory модели на инженерно оформленный memory stack.
