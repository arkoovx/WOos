# WoOS Releases

Этот документ описывает, как в WoOS устроен релизный контур и что обязан сделать автор изменений перед PR.

## 1. Базовое правило

В репозитории действует правило: **каждый PR = новая версия**.

Это означает, что перед отправкой PR нужно:
- обновить `VERSION`;
- добавить секцию в `CHANGELOG.md`;
- актуализировать затронутую документацию;
- убедиться, что образ `os.img` собирается.

## 2. Что попадает в версию

Обычно версия повышается, если PR:
- добавляет новую подсистему или заметный runtime/path;
- меняет поведение boot/kernel/UI;
- исправляет значимую регрессию;
- формализует важный архитектурный или процессный контракт.

## 3. Подготовка PR

Рекомендуемый чеклист:
1. Изучить связанные документы (`README.md`, roadmap, профильные docs).
2. Внести изменение небольшим инкрементом.
3. Обновить `VERSION`.
4. Добавить секцию текущей версии в `CHANGELOG.md`.
5. Обновить затронутые документы.
6. Проверить сборку `make os.img`.
7. В описании PR кратко указать:
   - что изменено;
   - как проверялось;
   - какой логичный следующий шаг.

## 4. Автоматическая публикация релиза

После merge PR workflow релиза:
1. проверяет, что у PR есть минимум один `APPROVED` review;
2. собирает `os.img` из merge-коммита;
3. читает версию из `VERSION`;
4. извлекает описание из соответствующей секции `CHANGELOG.md`;
5. создаёт тег `vX.Y.Z` и GitHub Release;
6. прикладывает к релизу собранный `os.img`.

## 5. Что ломает релизный pipeline

Наиболее типовые ошибки:
- версия в `VERSION` обновлена, но секция в `CHANGELOG.md` не добавлена;
- тег `vX.Y.Z` уже существует;
- PR не получил `APPROVED` review;
- сборка `os.img` перестала проходить.

## 6. Что должно оставаться синхронным

Перед PR держите в согласованном состоянии:
- `VERSION`;
- `CHANGELOG.md`;
- пользовательскую документацию (`README.md`);
- профильные документы в `docs/`;
- UI-строку версии, если она отображается в runtime.
