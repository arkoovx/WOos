# WoOS Releases

Этот документ объясняет **как читать релизный контур WoOS по файлам** и где находится источник истины для каждого аспекта процесса.
Он намеренно короче `RELEASE_CHECKLIST.md`: checklist остаётся обязательной памяткой, а этот файл служит навигацией по связанным артефактам.

## 1. С чего начинать

Если вы готовите PR или релиз, читайте документы в таком порядке:
1. `README.md` — общее правило проекта: каждый PR должен оставлять репозиторий в согласованном состоянии.
2. `RELEASE_CHECKLIST.md` — обязательные действия перед PR.
3. `.github/pull_request_template.md` — как оформить описание PR.
4. `.github/workflows/build.yml` — что именно проверяется CI-сборкой.
5. `.github/workflows/release.yml` — как публикуется релиз после merge.
6. `.github/scripts/extract_changelog.sh` — как из `CHANGELOG.md` извлекаются release notes.

## 2. Где находится источник истины

Чтобы не плодить дублирование, используйте такую модель:
- правило **«каждый PR = новая версия»** и обязательные шаги перед PR — в `RELEASE_CHECKLIST.md`;
- формат описания PR — в `.github/pull_request_template.md`;
- текущая версия — в `VERSION`;
- публичный список изменений — в `CHANGELOG.md`;
- автоматическая логика сборки и публикации — в `.github/workflows/` и `.github/scripts/`;
- этот документ — навигация по процессу и связи между файлами.

## 3. Как связаны ключевые файлы

### `VERSION`
Хранит номер текущей версии WoOS.
Любой PR обязан синхронизировать его с changelog и, если версия показывается пользователю, с runtime UI.

### `CHANGELOG.md`
Содержит публичную историю изменений по версиям.
Именно из него релизный workflow берёт текст release notes для соответствующего тега.

### `RELEASE_CHECKLIST.md`
Это главный operational-чеклист автора PR.
Если нужно изменить обязательные действия перед PR, в первую очередь правьте этот файл.

### `.github/pull_request_template.md`
Это интерфейс для автора PR.
Он не должен содержать отдельные правила, которые противоречат `RELEASE_CHECKLIST.md`; шаблон должен лишь направлять автора по уже зафиксированному процессу.

### `.github/workflows/build.yml` и `.github/workflows/release.yml`
Это исполняемая часть процесса.
Если документация обещает шаг, которого нет в workflow, источник несоответствия нужно исправлять сразу.

## 4. Когда обновлять этот документ

Обновляйте `docs/RELEASES.md`, если:
- меняется порядок чтения релизной документации;
- появляются новые process-файлы или workflow-скрипты;
- меняется распределение ответственности между `RELEASE_CHECKLIST.md`, шаблоном PR и GitHub Actions.

Если меняются сами обязательные шаги перед PR, вместе с этим документом нужно обновлять и `RELEASE_CHECKLIST.md`.
