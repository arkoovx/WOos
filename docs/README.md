# WoOS Docs Map

Этот файл — точка входа в документацию репозитория.
Его задача — не дублировать содержимое остальных документов, а направлять читателя по правильному порядку чтения.

## Как читать документацию WoOS

### 1. Начать с общего контекста
1. `README.md` — что уже умеет WoOS, как собрать образ и как запустить его локально.
2. `DEVELOPMENT_PLAN.md` — что уже сделано по roadmap и какой следующий крупный этап у проекта.

### 2. Понять устройство системы
3. `docs/ARCHITECTURE.md` — слои системы, init-flow, межмодульные контракты и архитектурные инварианты.

### 3. Углубиться в нужное направление
4. `docs/MEMORY.md` — читать, если вы работаете с `pmm`, будущим `vmm`, page tables, heap growth или MMIO mapping policy.
5. `docs/DEBUGGING.md` — читать, если у вас регрессия boot/runtime, нестабильная сборка или непонятно, как локализовать сбой.
6. `docs/RELEASES.md` — читать, если вы готовите PR, релиз или изменяете процесс версионирования/автопубликации.

## Какой файл является источником истины

Чтобы документы не расходились и не дублировали друг друга, в репозитории действует такая схема:
- `README.md` — краткий входной обзор и команды быстрого старта.
- `DEVELOPMENT_PLAN.md` — статус roadmap и приоритеты развития.
- `docs/ARCHITECTURE.md` — архитектурный контракт проекта.
- `docs/MEMORY.md` — специализированный roadmap memory stack.
- `docs/DEBUGGING.md` — практическая диагностика и safe fallback-пути.
- `RELEASE_CHECKLIST.md` — обязательный чеклист для каждого PR.
- `docs/RELEASES.md` — карта релизного процесса и связи между `VERSION`, `CHANGELOG`, checklist и GitHub workflows.
- `.github/pull_request_template.md` — шаблон описания PR, который опирается на release-процесс.

## Если меняете документацию

Используйте простой принцип:
- меняете общий обзор → правьте `README.md`;
- меняете план и приоритеты → правьте `DEVELOPMENT_PLAN.md`;
- меняете архитектурный контракт → правьте `docs/ARCHITECTURE.md`;
- меняете memory roadmap → правьте `docs/MEMORY.md`;
- добавляете новый типовой сценарий сбоя → правьте `docs/DEBUGGING.md`;
- меняете релизный процесс → правьте `RELEASE_CHECKLIST.md`, `docs/RELEASES.md` и при необходимости `.github/`-файлы.
