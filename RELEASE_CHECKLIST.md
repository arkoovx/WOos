# WoOS Release Checklist (обязательно для каждого PR)

> В этом репозитории **каждый PR = новая версия WoOS**.

## Что обязан сделать автор PR
1. Поднять версию в файле `VERSION` (пример: `1.0.2 -> 1.1.0`).
2. Добавить запись в `CHANGELOG.md` с кратким и понятным списком изменений.
3. Обновить затронутую документацию (`DEVELOPMENT_PLAN.md`, `README.md`, технические заметки и т.д.).
4. Проверить, что версия отображается в UI/выводе системы (если применимо).
5. Убедиться, что сборка проходит (`make os.img`).

## Что должен содержать PR
- Новый номер версии.
- Список изменений (что добавлено/изменено/исправлено).
- Обновления документации и пояснения для других разработчиков.
- Краткий план следующего шага (что делать в следующем PR).

## Политика тегов и автопубликации релизов
1. Релиз публикуется автоматически после merge PR.
2. Workflow `release-woos` автоматически:
   - собирает `os.img` из merge-коммита PR,
   - определяет версию из `VERSION` и извлекает секцию `## X.Y.Z` из `CHANGELOG.md`,
   - создаёт GitHub Release и тег `vX.Y.Z`, прикладывая образ и описание изменений.
3. Если тег `vX.Y.Z` уже существует или нет секции нужной версии в `CHANGELOG.md`, workflow завершится ошибкой для предотвращения неконсистентного релиза.
